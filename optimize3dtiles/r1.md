那你现在这个问题就已经进入**“真正的GPU几何吞吐瓶颈”阶段**了，不是 Cesium 组织结构的问题，也不是 DrawCall 的问题。

我们把结论先讲清楚：

> **DrawCall / Primitive 已经优化后仍然低帧率 ⇒ 基本就是“每帧要处理的三角形太多（Fill Rate / Vertex Throughput / Overdraw）”。**

---

# 一、你现在卡的本质是什么

如果你已经做到：

```text id="a1"
DrawCall ↓↓↓
Primitive ↓↓↓
Tile结构也优化过
```

但仍然卡，那么瓶颈只剩三类：

---

## ① 顶点吞吐（Vertex Bottleneck）

GPU在干这个：

```text id="a2"
transform(5000万顶点)
投影
裁剪
光栅化准备
```

哪怕三角形没被“真正画出来”，也要处理顶点。

---

## ② 光栅化压力（Triangle Fill）

例如：

```text id="a3"
屏幕 4K
+ 大面积近距离模型
```

每帧要：

```text id="a4"
光栅化几百万三角形
```

---

## ③ Overdraw（最隐蔽杀手）

地质模型非常典型：

```text id="a5"
多层岩体重叠
封闭体
剖切面
```

同一个像素可能被：

```text id="a6"
5~20个三角形重复绘制
```

GPU在做：

```text id="a7"
fragment shader 执行10次
最后只显示1次
```

---

# 二、关键判断：你到底是哪种瓶颈？

你现在必须先确认一个关键指标：

### 👉 GPU是不是满了

用 Chrome：

```text id="b1"
Performance → GPU Usage
```

或者：

```javascript id="b2"
viewer.scene.debugShowFramesPerSecond = true;
```

再配合：

```javascript id="b3"
viewer.scene.frameState.commandList.length
```

---

### 判断逻辑：

#### 情况A：GPU 90~100%

👉 纯 GPU 渲染瓶颈（最常见）

---

#### 情况B：CPU高

👉 Cesium Tile traversal / JS瓶颈

---

#### 情况C：GPU不满但帧率低

👉 **Overdraw / fill-rate 问题**

---

# 三、你这个项目最可能的真实问题（重点）

结合你做的是：

```text id="c1"
工程地质 + 3D Tiles + 封闭体 + 剖切
```

我可以很明确说：

> 👉 **80%概率是 Overdraw + Fill Rate 爆炸**

原因：

### 地质模型天然特性

```text id="c2"
多层封闭结构
密集三角网
大量重叠面
```

---

### Cesium默认渲染特点

```text id="c3"
不做早期深度剔除优化（类似游戏引擎那种）
```

---

# 四、为什么你“降 drawcall 没用”

因为你优化的是：

```text id="d1"
CPU提交次数
```

但你现在卡的是：

```text id="d2"
GPU真实在画多少像素
```

---

# 五、真正有效的优化方向（重点）

下面这些才是你下一阶段必须做的：

---

# ① Depth Pre-Pass（关键）

先画深度，不画颜色：

```text id="e1"
第一遍：只写 depth
第二遍：正常渲染
```

效果：

```text id="e2"
减少 overdraw 50%~90%
```

---

Cesium默认没有很好利用这一点。

---

# ② Early-Z 强制优化（非常关键）

确保：

```text id="e3"
opaque geometry
```

全部：

```text id="e4"
不开 blending
不开 discard
```

---

否则：

```text id="e5"
early-z失效
```

---

# ③ Tile 级“屏幕误差裁剪”（比LOD更重要）

你现在可能：

```text id="f1"
Tile都加载了
```

但实际上：

```text id="f2"
屏幕只需要 5%
```

---

必须做：

```text id="f3"
screen-space error 提前截断
```

甚至可以更激进：

```text id="f4"
距离 + 屏幕面积联合裁剪
```

---

# ④ 几何简化（不是LOD，是“结构简化”）

你现在的问题不是：

```text id="g1"
LOD不够
```

而是：

```text id="g2"
每个Tile内部太密
```

---

真正有效的是：

### Cluster simplification（类似 Nanite）

```text id="g3"
三角形聚类
↓
局部重建
↓
减少局部密度
```

---

# ⑤ GPU Instancing（如果有重复地层）

如果存在：

```text id="g4"
重复岩层
重复地质单元
```

必须：

```text id="g5"
Instancing
```

---

# ⑥ 最关键的一条（很多人忽略）

## 👉 你的瓶颈可能根本不是“总三角形数量”，而是“屏幕三角形数量”

举个极重要的区别：

| 指标         | 是否重要  |
| ---------- | ----- |
| 总三角形       | ❌不关键  |
| 屏幕覆盖三角形    | ✅关键   |
| overdraw次数 | 🚨最关键 |
